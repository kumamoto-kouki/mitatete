例① 品質の点検 完了

計画プロダクトを点検・検証・記録する計画を立てる
実行実装と記録のあと、別の担当役（独立チェック）で再点検
確認チェック役が 本物のバグ（chat.ts）・通信制限の未設定・仕様の食い違いを発見
改善3件を修正 → 再テストで 全合格

例② 作業の任せ方の試行 完了

計画新コマンドの実装を別の作業役（AI）に任せてみる
実行作業用の別コピーで実装し、専用ブランチに記録
確認別のチェック役が 合格 判定。 取りまとめ役も80件のテストを自分で実行し直して確認
改善本体に統合し、つまずいた点を運用ルールに反映

AI にどこまで任せるか（3段階）段階を上げるほど自動、下げるほど人が関与

段階1: 報告だけ

状況を仕分け（優先度づけ）してメモに残す。コードは変えない。

段階2: 補助 いまここ

作る役とチェックする役を分けて修正。本体への取り込み・公開は人が判断。

段階3: 無人

禁止事項以外を自動。予算・計測・禁止リストを整えてから検討（当面は段階2まで）。

暴走を防ぐ仕組み

  • 作る役とチェックする役を分ける: 自分で作って自分で合格を出すと甘くなるため、別の役が点検。テスト結果の提示を必須にし「やった体」の偽装を防ぐ。
  • 止めどきを最初に決める: 目標達成／人へ引き継ぎ／試行回数の上限／予算超過。
  • 使う費用に上限: 軽い作業は任せず直接やる（費用と見えやすさのバランス）。
  • 中身の理解を置き去りにしない: 節目ごとに要約と読み直しを行い、AI に作業量を渡すのでなく「判断」を渡す。
  • 同時作業の衝突を防ぐ: 作業用の別コピーに分けて進める。

用語集（固有名詞のかんたん解説）③ふだんの言葉で言い換え

Tauri（タウリ）
Web 技術でパソコン用アプリを作る仕組み。本アプリの土台。
spec（スペック）
機能ごとの仕様書。要件→設計→タスクの順に固めていく。
E2E テスト
End to End＝端から端まで。アプリを実際に起動して通しで動くか確かめる自動テスト。
PDCA
計画(Plan)→実行(Do)→確認(Check)→改善(Act)のくり返し。
CSP
Content Security Policy。アプリの通信先や読み込み先を必要な範囲に絞るセキュリティ設定。
semgrep（セムグレップ）
コードの危険な書き方や脆弱性を自動で探すスキャナー。
happy-dom / insta / WebdriverIO
テスト用の道具。順に「画面部品の検査」「出力の固定検査」「実アプリ操作」。
worktree（ワークツリー）
リポジトリの作業用の別コピー。同時編集の衝突を避けるために使う。
push（プッシュ）
手元の変更を共有サーバー（GitHub）へ反映＝公開する操作。
コンダクター・オーケストレーション
本プロジェクトの進め方の呼び名。取りまとめ役が作業を任せ、別の役がチェックし、証拠で受け入れる体制。
